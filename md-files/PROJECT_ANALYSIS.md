## 1. Information Architecture

### 1.1 نظرة عامة على المنتج

**Dasm inspection** منصة رقمية لربط مالكي السيارات بالورش لإجراء فحوصات دورية وإدارة المواعيد. المنتج يخدم ثلاث شخصيات:

- **العميل (Customer):** إدارة سياراته، متابعة تقارير الفحص، حجز المواعيد.
- **الورشة (Workshop):** استقبال المواعيد، تنفيذ الفحص، رفع التقارير والصور.
- **الإداري (Admin):** مراقبة النظام، إدارة الورش والمستخدمين، التقارير والإحصائيات.

### 1.2 هيكل الشاشات المستنتج من Figma

#### تطبيق العميل
- شاشة الترحيب / Onboarding
- تسجيل الدخول / Login
- الشاشة الرئيسية / Home Dashboard
- ملف السيارة / Car Profile
- تقرير الفحص / Inspection Report
- حجز موعد / Book Appointment
- مشاركة QR / QR Share
- الإشعارات / Notifications
- إضافة سيارة / Add Car
- الاشتراكات / Subscription
- الملف الشخصي / Profile

#### تطبيق الورشة
- الرئيسية / Workshop Dashboard
- قائمة المواعيد / Appointments
- نموذج الفحص / Inspection Form
- تفاصيل الفحص / Inspection Details
- تاريخ السيارة / Car History
- إدارة الصور / Photos

#### لوحة الإدارة
- لوحة تحكم إحصائية
- إدارة الورش
- إدارة المستخدمين
- مراقبة الفحوصات
- التقارير والمدفوعات
- إدارة الاشتراكات

### 1.3 علاقات الكيانات

```
User ←→ Cars (واحد لمتعدد)
Car ←→ Inspections (واحد لمتعدد)
Car ←→ Appointments (واحد لمتعدد)
Workshop ←→ Inspections
Workshop ←→ Appointments
User ←→ Subscriptions
User ←→ Notifications
```

## 2. User Flows

### 2.1 مسار العميل الأساسي

1. فتح التطبيق → Onboarding (أول مرة)
2. تسجيل الدخول / إنشاء حساب
3. الشاشة الرئيسية → قائمة السيارات
4. اختيار سيارة → ملف السيارة
5. عرض آخر تقرير فحص
6. حجز موعد جديد → اختيار ورشة وتاريخ
7. استلام إشعار بالموافقة
8. بعد الفحص → عرض التقرير المحدَّث

### 2.2 مسار الورشة الأساسي

1. تسجيل الدخول
2. لوحة التحكم → قائمة المواعيد اليوم
3. اختيار موعد → بدء الفحص
4. تعبئة نموذج الفحص (بنود متعددة)
5. رفع صور للأجزاء
6. إنهاء الفحص وإصدار التقرير
7. إرسال التقرير للعميل

### 2.3 مسار الإداري

1. تسجيل الدخول
2. لوحة التحكم الإحصائية
3. التنقل بين: الورش، المستخدمين، الفحوصات، التقارير
4. عمليات التحقق والموافقة
5. تصدير التقارير

## 3. UI Components Breakdown

### 3.1 مكونات التنقل

- **Bottom Navigation:** تبويبات ثابتة (Home, Cars, Appointments, Notifications, Profile)
- **Top App Bar:** شعار، عنوان، أيقونات إجراءات
- **Tabs:** تنقل أفقي بين أقسام (مثلاً: جميع المواعيد / قادمة / مكتملة)

### 3.2 مكونات العرض

- **Cards:** بطاقات السيارات، بطاقات المواعيد، بطاقات تقرير الفحص
- **Lists:** قوائم عناصر مع أيقونة، عنوان، وصف ثانوي
- **Tables:** جداول للإدارة (ورش، مستخدمين، فحوصات)
- **Badges:** شارات الحالة (pending, completed, cancelled)
- **Progress indicators:** شريط تقدم الفحص أو الـ onboarding

### 3.3 مكونات الإدخال

- **Text Inputs:** حقول نصية للنماذج (البريد، الهاتف، بيانات السيارة)
- **Buttons:** Primary, Secondary, Outline, Icon
- **Checkboxes / Radio:** اختيار خيارات الفحص
- **Date Picker:** اختيار تاريخ الموعد
- **File/Image Upload:** رفع صور أجزاء السيارة
- **Dropdown / Select:** اختيار الورشة، الماركة، الطراز

### 3.4 مكونات خاصة

- **QR Code Display:** لعرض/مشاركة بيانات السيارة
- **Inspection Score:** عرض درجة الفحص (مثلاً دائرة أو شريط)
- **Car Avatar:** صورة أو أيقونة السيارة مع معلومات مختصرة
- **Rating Stars:** تقييم الورشة
- **Status Chips:** حالة الموعد أو الفحص

## 4. Design System

### 4.1 نظام الألوان

من خلال تحليل Figma، يتم التمييز بين الأدوار بألوان أساسية:

| الدور | اللون الأساسي | الاستخدام |
|------|---------------|-----------|
| العميل | أزرق (#2563EB تقريباً) | Primary buttons, links, accents |
| الورشة | بنفسجي (#7C3AED تقريباً) | نفس الاستخدام لتطبيق الورشة |
| الإدارة | أحمر/وردي (#DC2626 تقريباً) | لوحة الإدارة |

ألوان دلالية مقترحة:

- **Success:** أخضر (اكتمال الفحص، موعد مؤكد)
- **Warning:** برتقالي (تحذير، انتهاء قريب)
- **Error:** أحمر (إلغاء، خطأ، حرج)
- **Neutral:** رمادي للتدرجات والخلفيات

### 4.2 Typography

- **العناوين:** خط عريض، أحجام متدرجة (مثلاً 24px للعنوان الرئيسي، 18px للفرعي)
- **النص العادي:** حجم قابل للقراءة على الموبايل (14–16px)
- **التسميات:** أصغر حجماً، لون ثانوي
- **دعم RTL:** ضروري للنصوص العربية

### 4.3 المسافات والتخطيط

- **Spacing scale:** 4px, 8px, 16px, 24px, 32px
- **Padding للبطاقات:** 16px أفقي، 12–16px عمودي
- **Gap بين العناصر:** 8–12px في القوائم
- **هوامش الشاشة:** 16–24px على الجوانب

### 4.4 Border Radius & Shadows

- **Cards:** border-radius صغير أو متوسط (4–8px)
- **Buttons:** نفس النطاق
- **Shadows:** خفيفة للبطاقات، أوضح للـ modals

## 5. Strengths

- **فصل واضح للأدوار:** ثلاث تجارب منفصلة (عميل، ورشة، إدارة) مع تدفق منطقي لكل دور.
- **تغطية وظيفية واسعة:** من Onboarding حتى تقارير الفحص والاشتراكات.
- **مكونات قابلة لإعادة الاستخدام:** أزرار، بطاقات، حقول، تنقل متسق.
- **دعم الموبايل:** تخطيط يناسب الشاشات الصغيرة مع Bottom Nav.
- **نظام ألوان مميز:** تمييز بصري بين الأدوار يساعد في الهوية البصرية.
- **مسارات مستخدم منظمة:** تدفقات واضحة للعميل والورشة والإداري.

## 6. Weaknesses / Opportunities

### 6.1 Weaknesses

- **تعقيد نموذج الفحص:** يحتاج تقسيم إلى خطوات (Step Wizard) لتحسين تجربة الورشة.
- **كثافة الشاشات:** بعض الشاشات قد تكون مزدحمة؛ تحتاج تبسيط هرم بصري.
- **غموض بعض الحالات:** حالات مثل "قيد المراجعة" أو "معلق" تحتاج تعريف واضح في النصوص.
- **الـ Empty States:** تحتاج تصميم صريح عندما لا توجد سيارات أو مواعيد.
- **الـ Error States:** رسائل الخطأ وتصميم حالات الفشل تحتاج وضوحاً أكبر.

### 6.2 Opportunities

- **استخدام Design Tokens:** تصدير الألوان والمسافات والخطوط كـ tokens لتنفيذ موحد.
- **توحيد المكونات:** مكونات مشتركة مع `role` prop بدلاً من نسخ لكل تطبيق.
- **تحسين Accessibility:** ARIA، تباين ألوان، دعم القارئ الشاشي.
- **Dark Mode:** إضافة ثيم داكن للوضع الليلي.
- **تحسين أداء الصور:** lazy loading وضغط للصور المرفوعة.

## 7. Frontend Implementation Strategy

### 7.1 تقسيم الواجهة إلى Components

#### Shared (مشتركة)
- `AppShell` — الإطار العام مع Top Bar و Bottom Nav
- `Card`, `Button`, `Input`, `Select`, `Badge`
- `LoadingSpinner`, `EmptyState`, `ErrorState`
- `PhoneFrame` — إطار شاشة الموبايل (للـ preview إذا لزم)

#### Customer
- `CarCard`, `CarAvatar`, `InspectionSummaryCard`
- `AppointmentCard`, `SubscriptionCard`
- `QRCodeDisplay`

#### Workshop
- `AppointmentListItem`, `InspectionFormWizard`
- `InspectionItemRow`, `PhotoUploadGrid`
- `CarHistoryTimeline`

#### Admin
- `StatsCard`, `DataTable`, `UserRow`, `WorkshopRow`
- `InspectionStatusBadge`, `ChartPlaceholder`

### 7.2 صفحات (Pages / Routes)

- تجميع الشاشات تحت مسارات واضحة: `/customer/*`, `/workshop/*`, `/admin/*`
- استخدام Layout مشترك لكل دور مع تبديل المحتوى فقط
- Lazy loading للصفحات لتحسين الأداء

### 7.3 State Management

- **Server State:** React Query أو SWR للبيانات من الـ API
- **Client State:** Zustand أو Context للحالة المحلية (مثل الـ role، الـ theme)
- **Form State:** React Hook Form مع Zod للتحقق

## 8. الخلاصة

التصميم في Figma يغطي رؤية المنتج ومسارات المستخدم بشكل جيد. التركيز في المرحلة التالية يجب أن يكون على:

1. تحويل التصميم إلى Design Tokens.
2. إنشاء هيكل مشروع Next.js مع المجلدات والمكونات.
3. تنفيذ المكونات الأساسية ثم ربطها بالشاشات.
4. ربط الواجهة بالـ Backend عند جاهزية الـ API.

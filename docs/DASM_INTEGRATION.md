# DASM Integration — Assumptions

## مبدأ

- **dasm-inspection** ريبو مستقل؛ لا يُبنى بديل لمنصة DASM (مزادات، ملفات مستخدمين، إدارة مركبات).
- التطبيق يفترض أن **مصدر الحقيقة** للمركبة والمستخدم هو DASM، ويخزن فقط **مراجع** (`dasm_car_id`, `dasm_user_id`).

## إنشاء طلب فحص (مستقبل)

1. حدث في DASM (مثلاً بعد شراء أو جدولة) يستدعي API هذا المنتج **أو** واجهة إدارية تنشئ الطلب مع المرجع.
2. التحقق من صلاحية المستخدم عبر JWT موقّع من DASM أو بوابة موحدة (SSO) — غير مطبق في V1.

## البيانات المشتركة

| من DASM | الاستخدام هنا |
|---------|----------------|
| معرف المركبة | `dasm_car_id` + `vehicleLabel` للعرض (قد يُجلب الاسم من DASM لاحقاً) |
| معرف المستخدم | `dasm_user_id` للعميل/المشتري عند الحاجة |
| مرجع مزاد | `auction_reference` اختياري |

## API موصى به لاحقاً

- `POST /inspection-requests` — body يتضمن `dasm_car_id`, سياق اختياري.
- Webhooks أو polling لتحديث حالة الطلب في واجهات DASM — قرار منتج لاحق.

## ما لا نفعله

- لا مزايدات، لا محفظة، لا استبدال لوحة تحكم DASM الرئيسية — فقط **خدمة فحص/ورش**.

## Supabase (V1)

- الطلبات والتقارير والمرفقات وسجل الحالات تُخزَّن في Postgres (`inspection_*`) — **دون** جداول أو منطق شحن.
- المتغيرات: `NEXT_PUBLIC_SUPABASE_URL`، `SUPABASE_SERVICE_ROLE_KEY` (خادم فقط، انظر `frontend/.env.example`).
- إنشاء طلب من الواجهة يحفظ `dasm_car_id` كمرجع دون استدعاء منصة DASM حتى يُربط الـ API لاحقاً.
